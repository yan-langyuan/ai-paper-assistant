## 二、需求拆解（总-分结构）

### 2.0 产品模块总览

```
AI论文助手
├── 模块一：智能文献助手（U01/U02/U03/U11/U12 + B01/B02/B09）
├── 模块二：格式排版助手（U04/U05/U13 + B10/B11）
```

### 2.1 模块一：智能文献助手

#### M1-01 文献导入与解析

| 字段 | 内容 |
|------|------|
| **需求编号** | M1-01 |
| **关联需求** | U01, B09 |
| **功能描述** | 用户通过上传PDF文件、粘贴链接或输入标题，系统解析文献并提取元数据 |
| **需求说明** | **场景**：用户手头有10篇课程论文PDF，希望快速了解每篇的核心内容。<br>**操作逻辑**：①用户拖拽/选择PDF上传（支持批量10篇同时上传）→ ②系统解析PDF文本，提取标题、作者、年份、期刊、DOI → ③若DOI存在，调用知网/万方API补全元数据 → ④文献存入"我的文献库"，显示解析状态（解析中/已完成/解析失败）。<br>**预期效果**：单篇解析<15秒，元数据提取准确率>90%，支持中英文PDF。 |
| **边界说明** | 仅做解析和元数据提取，不做全文翻译。不支持扫描版PDF（OCR为后续迭代需求）。最大单篇文件50MB。 |
| **验收标准** | 1. 上传1篇PDF，15秒内完成解析并展示标题/作者/年份/期刊/DOI<br>2. 批量上传10篇PDF，30秒内全部解析完毕<br>3. 输入知网论文链接，自动抓取元数据<br>4. 解析失败时提示"该文献暂不支持解析"，不阻塞其他文献 |

#### M1-02 文献结构化摘要生成

| 字段 | 内容 |
|------|------|
| **需求编号** | M1-02 |
| **关联需求** | U01 |
| **功能描述** | 基于文献全文，AI自动生成结构化摘要卡片 |
| **需求说明** | **场景**：用户不想读完整篇20页论文，希望快速知道"这篇文献研究了什么问题、用了什么方法、得出什么结论"。<br>**操作逻辑**：①M1-01解析完成后，自动触发摘要生成 → ②AI按模板输出：研究背景（1句）- 研究问题（1句）- 研究方法（1句）- 核心发现（1-3点）- 研究局限（1句）→ ③摘要卡片展示在文献详情页，支持复制、导出。<br>**预期效果**：摘要生成<30秒/篇，用户可1分钟内判断文献是否值得精读。 |
| **边界说明** | 摘要为AI生成内容，**必须标注"AI生成摘要，仅供参考，请核实原文"**。不替代全文阅读，不做"这篇值得读/不值得读"的二元判断。 |
| **验收标准** | 1. 上传1篇20页PDF，30秒内生成结构化摘要卡片<br>2. 摘要包含研究问题-方法-发现-局限四个模块<br>3. 摘要底部有"AI生成，请核实原文"提示<br>4. 支持一键复制摘要文本 |

#### M1-03 核心观点自动提炼

| 字段 | 内容 |
|------|------|
| **需求编号** | M1-03 |
| **关联需求** | U02 |
| **功能描述** | 在文献全文中自动标记可引用段落，标注引用价值 |
| **需求说明** | **场景**：用户已读完文献，开始写文献综述，需要快速找到"哪些段落适合引用"。<br>**操作逻辑**：①用户点击"标记可引用段落"→ ②AI分析全文，高亮标注：核心定义句（蓝色）、关键数据/结论（绿色）、可引用的研究方法描述（黄色）→ ③每个标注附带标签（如"可引入文献综述-概念界定部分"）→ ④用户点击标注可跳转原文对应位置。<br>**预期效果**：平均每篇文献标记5-10个可引用段落，标记准确率>85%。 |
| **边界说明** | 标记为"建议"性质，最终引用判断由用户决定。不做文献质量评分（避免引导性偏见）。 |
| **验收标准** | 1. 打开已解析的文献，点击"标记可引用段落"，3秒内完成标注<br>2. 标注段落用不同颜色区分定义/数据/方法三类<br>3. 每个标注有对应的引用场景标签<br>4. 点击标注能跳转到PDF原文对应位置 |

#### M1-04 多文献观点对比

| 字段 | 内容 |
|------|------|
| **需求编号** | M1-04 |
| **关联需求** | U03 |
| **功能描述** | 选择2-5篇文献，AI自动生成观点对比矩阵 |
| **需求说明** | **场景**：用户写文献综述时，需要对比5篇文献对同一个概念的不同定义。手动摘录对比耗时巨大。<br>**操作逻辑**：①用户从文献库勾选2-5篇文献 → ②点击"对比分析"→ ③AI生成对比矩阵，行=对比维度（研究问题/方法/核心结论/样本/局限），列=各文献 → ④支持切换对比维度（如只看"方法论对比"）→ ⑤对比表可导出为表格。<br>**预期效果**：生成对比矩阵<60秒，用户10分钟内完成原本2小时的文献梳理工作。 |
| **边界说明** | 最多支持5篇同时对比（超过5篇AI质量下降明显）。对比维度的结论为AI提取，可能存在偏差，标注"AI分析，建议核实"。 |
| **验收标准** | 1. 勾选3篇文献，点击对比，60秒内生成对比矩阵<br>2. 矩阵至少包含5个对比维度<br>3. 支持切换单个维度查看详情<br>4. 对比表支持复制为Markdown表格 |

#### M1-05 真实文献检索与溯源

| 字段 | 内容 |
|------|------|
| **需求编号** | M1-05 |
| **关联需求** | U12, B01, B02 |
| **功能描述** | 根据用户选题，从知网/万方检索真实文献，提供可溯源的文献列表 |
| **需求说明** | **场景**：用户输入选题"社交媒体对大学生心理健康的影响"，系统检索出相关真实文献，而非AI编造的假文献。<br>**操作逻辑**：①用户输入选题关键词 → ②系统调用知网/万方API检索 → ③返回真实文献列表，每条显示标题、作者、年份、期刊、引用量、DOI → ④每条文献标注"数据来源：知网"、"DOI可验证" → ⑤用户可将文献加入"我的文献库"。<br>**预期效果**：文献真实率100%（均来自数据库，非AI生成）。检索响应<5秒。 |
| **边界说明** | 该功能依赖知网/万方API开放程度。若API不可用，降级为"推荐检索关键词+数据库链接"。不做"AI猜文献"——这是竞品虚构文献的根源。 |
| **验收标准** | 1. 输入选题关键词，返回20条+真实文献，每条有DOI<br>2. 点击DOI链接可在知网验证<br>3. 无任何AI虚构文献混入<br>4. 检索失败时明确提示"数据库暂不可用"并给出手动检索引导 |

### 2.2 模块二：格式排版助手

#### M2-01 参考文献一键格式化

| 字段 | 内容 |
|------|------|
| **需求编号** | M2-01 |
| **关联需求** | U04, B11 |
| **功能描述** | 用户输入文献信息（或从文献库选择），一键生成符合指定标准的参考文献条目 |

| **验收标准** | 1. 输入一篇期刊论文信息，选择GB/T 7714，正确生成格式条目<br>2. 选择30篇文献，一键导出Word，顺序按拼音排列<br>3. 切换APA标准，英文文献格式自动切换<br>4. 输入DOI，自动补全作者/标题/期刊/年份信息 |

| **关联需求** | U05, B10 |

### 2.3 模块三：降重改写助手

#### M3-01 段落重复检测与高亮

| 字段 | 内容 |
|------|------|
| **需求编号** | M3-01 |
| **关联需求** | U06, B03 |
| **功能描述** | 用户粘贴论文段落，系统检测潜在重复并高亮标注 |
| **需求说明** | **场景**：用户论文某段被导师指出重复率高，但不知道具体哪里重复、怎么改。<br>**操作逻辑**：①用户粘贴需检测的文本段落（建议500-3000字/次）→ ②系统在本地进行语义比对，标记可能重复的句子（红色=高度相似>80%，黄色=中度相似50-80%，绿色=低风险<50%）→ ③点击高亮句，展开详情：相似度百分比、疑似重复来源描述 → ④系统给出"改写方向建议"（如"该句与常见表述相似，建议将被动语态改为主动语态，或更换论证角度"）。<br>**预期效果**：检测在本地完成，文本不上传服务器。处理2000字<5秒。 |
| **边界说明** | **关键差异化设计**：降重检测在本地完成，用户文稿绝不离开设备。不做实际查重（需对接知网/维普等查重数据库，V1.1补充）。相似度基于语言模型语义比对，非查重系统报告，**必须明确提示"本检测非正式查重结果"**。 |
| **验收标准** | 1. 粘贴2000字文本，5秒内完成检测并高亮标注<br>2. 高亮分为红/黄/绿三档，点击可查看详情<br>3. 每个高亮句附带改写方向建议<br>4. 页面明确提示"本检测为AI辅助判断，非正式查重结果"<br>5. 网线断开后功能仍可用（验证本地处理） |

#### M3-02 逐句改写建议

| 字段 | 内容 |
|------|------|
| **需求编号** | M3-02 |
| **关联需求** | U06, U07, B10 |
| **功能描述** | 针对高亮重复句，逐句生成多个改写方案，用户自主选择 |
| **需求说明** | **场景**：用户某句被判为高重复，系统给出3个改写方向，用户选择最喜欢的那个，再按自己的风格微调。**这是产品的核心差异功能——"建议而不替代"**。<br>**操作逻辑**：①用户点击某个高亮句子 → ②系统生成3个改写方案：方案A（保守：调整语序+替换非关键词）、方案B（均衡：改变句式结构+保留术语）、方案C（激进：重构句子+更换论证逻辑）→ ③每个方案标注"哪些词没变（术语保留）""哪些部分改了" → ④用户选择一个方向，在此基础上手动编辑 → ⑤改写后的句子，可重新检测重复度，形成"检测→改写→再检测"闭环。<br>**预期效果**：改写方案生成<3秒/句。用户采纳并再改后，重复度平均下降40-60%。 |
| **边界说明** | 核心设计约束：**不提供"一键改写全文"按钮**——这是与竞品（千笔AI）的关键差异。用户的学术判断不可被自动化替代。方案C（激进改写）附带提醒"该方案改动较大，请核实学术含义是否变化"。 |
| **验收标准** | 1. 点击高亮句，3秒内生成3个改写方案<br>2. 方案A/B/C改动程度递增，术语保留标记清晰<br>3. 用户编辑改写方案后可重新检测重复度<br>4. 无"一键改写全文"按钮<br>5. 方案C附带学术含义变更提醒 |

#### M3-03 术语保护机制

| 字段 | 内容 |
|------|------|
| **需求编号** | M3-03 |
| **关联需求** | U07 |
| **功能描述** | 改写过程中自动锁定专业术语，确保降重不"伤害"学术准确性 |
| **需求说明** | **场景**：用户论文中"异质性检验""DID模型""调节效应"等术语不可被改写。这是用户调研中"降重把专业术语改错"痛点的直接解决方案。<br>**操作逻辑**：①系统在改写前自动识别文本中的专业术语（基于学科术语库）→ ②术语在原文中用下划线标记 → ③用户可手动添加/移除保护词 → ④改写方案生成时，被保护的术语保持原样，只改写非术语部分 → ⑤若检测到术语可能被误改，系统发出警告。<br>**预期效果**：术语识别准确率>90%。改写后术语误改率<5%。 |
| **边界说明** | V1.0内置通用学术术语库（10万+词条）。学科专属术语库（如医学、法律）为V1.1需求。用户手动标记的术语优先级高于自动识别。 |
| **验收标准** | 1. 粘贴含"回归分析""显著性水平"等术语的文本，系统自动标记为保护词<br>2. 改写方案中保护词原样保留<br>3. 用户可手动添加/移除保护词<br>4. 术语被改写时弹出警告提示 |

#### M3-04 写作风险预警

| 字段 | 内容 |
|------|------|
| **需求编号** | M3-04 |
| **关联需求** | B08, B12 |
| **功能描述** | 监控用户对AI功能的依赖程度，适时发出学术诚信提醒 |
| **需求说明** | **场景**：用户连续使用AI改写超过30%的论文篇幅，系统在侧边栏弹出温和提醒。<br>**操作逻辑**：①系统统计用户当日/本周使用AI改写/生成的文字量占论文总篇幅的比例 → ②超过20%时，侧边栏出现浅黄色提示 → ③超过30%时，弹出温和提醒"AI生成内容占比较高，建议您在此基础上进行实质性修改和补充"→ ④超过50%时，系统在导出前强制弹出《学术诚信提醒》，用户确认后才可继续。<br>**预期效果**：平衡功能可用性与学术合规，降低用户误用风险。 |
| **边界说明** | 该功能为"提醒"而非"限制"，不阻止用户使用，仅做教育和告知。文案设计避免说教感（不用"警告""禁止"等措辞）。 |
| **验收标准** | 1. AI使用超20%时，侧边栏出现浅黄提示<br>2. 超30%时，弹出温和提醒弹窗<br>3. 超50%时，导出前强制弹出学术诚信确认<br>4. 提醒文案无说教感，用户确认后不二次拦截 |

## 五、需求验证记录

### 5.1 需求调整记录

| 原需求 | 问题 | 调整后 | 调整原因 |
|------|------|------|------|
| U04: 参考文献一键格式化 + 自动查重 | 功能耦合过紧 | 拆为 M2-01（格式化）+ M3-01（检测）两个独立模块 | 格式化和查重是不同场景，用户不总同时需要 |
| U06: "一键降重全文" | 与产品定位冲突 | 调整为 M3-02 "逐句改写建议"，不做一键全文改写 | 用户调研结论"不要替我写"，竞品一键改写已证明效果差且学术风险高 |
| M1-02 最初设计包含"文献推荐评分" | 引导性偏见风险 | 去掉评分，只做客观的结构化摘要 | 评分可能引导用户忽略"低分"但实际有价值的文献 |
| B05: 微信小程序作为MVP首发 | 开发成本过高 | 降为P3暂不做，MVP仅Web端 | MVP验证核心价值，过早多端投入ROI低 |
| U08: 预查重预估 | 准确度难保证 | 降为P3，M3-01仅做语义相似度检测并明确标注"非正式查重" | 正式查重需对接数据库，准确度不足会误导用户 |
